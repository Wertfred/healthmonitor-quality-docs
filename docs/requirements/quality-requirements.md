# Нефункциональные требования к сервису HealthMonitor

Проект **HealthMonitor** компании *SetCom* — это веб-сервис для сбора и анализа метрик здоровья приложений.  
Цель данного документа — определить измеримые требования к качеству работы системы.

---

### NFR-PERF-01
**Название:** Время ответа API эндпоинта сбора метрик.  
**Описание:** 95-й перцентиль времени ответа для POST /api/v1/metrics не должен превышать 200 мс.  
**Метрика и целевое значение:** p95 < 200 мс  
**Способ проверки:** Нагрузочное тестирование с помощью k6 или JMeter.

---

### NFR-REL-01
**Название:** Доступность сервиса.  
**Описание:** Сервис должен быть доступен для пользователей большую часть времени.  
**Метрика и целевое значение:** Uptime ≥ 99.5% (SLA)  
**Способ проверки:** Мониторинг через Prometheus + Grafana, отчёты uptime.

---

### NFR-SEC-01
**Название:** Аутентификация и безопасность данных.  
**Описание:** Все API-запросы должны выполняться только с валидным токеном и передаваться по HTTPS.  
**Метрика и целевое значение:** 100% запросов используют HTTPS; неавторизованные запросы → 401 Unauthorized  
**Способ проверки:** Автоматические API-тесты через Postman/Newman.

---

### NFR-MAINT-01
**Название:** Логирование и сопровождаемость.  
**Описание:** Все критические события должны логироваться с указанием уровня (INFO/WARN/ERROR) и trace-id.  
**Метрика и целевое значение:** 100% ошибок фиксируются в логах  
**Способ проверки:** Проверка логов в ELK-стеке (Elasticsearch + Kibana)
